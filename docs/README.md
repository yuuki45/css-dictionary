## ドキュメントの使い方

### 新しく参加する開発者向け

1. **[開発ガイドライン](./development-guide.md)** から始めて、開発環境をセットアップ
2. **[アーキテクチャドキュメント](./architecture.md)** でプロジェクト全体を理解
3. **[コンポーネント設計](./components.md)** でコンポーネント構造を把握
4. 開発開始

### 機能追加・修正を行う場合

1. **[データ構造ドキュメント](./data-structure.md)** で必要なデータ型を確認
2. **[コンポーネント設計](./components.md)** で既存コンポーネントを確認
3. **[ルーティング設計](./routing.md)** で新しいルートが必要か検討
4. **[開発ガイドライン](./development-guide.md)** に従って実装

### AI（Claude等）に開発を依頼する場合

**コンテキストエンジニアリング**のアプローチを採用し、関連ドキュメントを明示的に参照することで、一貫性のある高品質な実装が可能になります。

#### 基本的な依頼方法

1. **関連ドキュメントを明示的に参照**
   - 実装に関わるドキュメントのパスを提示
   - 特にarchitecture.md、components.md、data-structure.mdが重要

2. **明確な要件と期待する動作を説明**
   - 何を実装するか
   - どのように動作するべきか
   - 既存のどの機能と連携するか

3. **遵守すべき規約を指示**
   - コーディング規約（development-guide.md参照）
   - 既存のパターンとの一貫性
   - TypeScript型定義の適切な実装

#### 実践例

**例1: 新機能追加**
```
「docsフォルダのarchitecture.mdとcomponents.mdを参照して、
CSSプロパティの比較機能を追加してください。

要件：
- 複数のプロパティを選択して並べて比較できる
- お気に入り機能と連携する
- ルーティングは/compare/[ids]を使用

既存のコンポーネント設計パターンに従い、TypeScript型定義を
適切に行い、useFavoritesフックを活用してください。」
```

**例2: リファクタリング**
```
「development-guide.mdのコーディング規約に従って、
SearchBarコンポーネントをリファクタリングしてください。

components.mdを参照し、既存のコンポーネント設計パターンを
維持しながら、パフォーマンスを改善してください。」
```

**例3: データ構造の拡張**
```
「data-structure.mdを参照して、CSSPropertyに
difficulty（難易度）フィールドを追加してください。

既存のバリデーションルールに従い、型定義を更新し、
関連するコンポーネントも適切に更新してください。」
```

#### コンテキストエンジニアリングのベストプラクティス

✅ **推奨:**
- 複数の関連ドキュメントを参照
- 既存パターンとの一貫性を明示的に指示
- 型安全性とエラーハンドリングを要求
- レスポンシブ対応とダークモード対応を確認

❌ **避けるべき:**
- ドキュメントを参照せずに依頼
- 曖昧な要件や期待値
- 既存パターンを無視した実装
- ドキュメントの更新を忘れる
